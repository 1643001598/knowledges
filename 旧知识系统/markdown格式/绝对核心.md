              - WSL 1
                - 架构： 它是一个转换层，将 Linux 系统调用实时转换为 Windows 系统调用。
                - 优点：
                  - 与 Windows 文件系统的互操作性极佳，可以直接在 Linux 环境中访问 Windows 文件，反之亦然，速度很快。
                  - 启动速度快，占用资源少。
                - 缺点：
                  - 由于需要系统调用转换，I/O 密集型操作（如git克隆、编译项目）性能相对较慢。
                  - 不完全兼容所有 Linux 内核特性。
              - WSL 2
                - 架构： 它是一个轻量级虚拟机，使用微软优化的 Hyper-V 虚拟化技术在 Windows 上运行一个真正的 Linux 内核。
                - 优点：
                  - 完整的系统调用兼容性，可以运行 Docker 等更多应用。
                  - 性能大幅提升，尤其是在文件 I/O 和运行 Linux 软件方面，速度接近原生 Linux。
                  - 保留了与 Windows 文件系统的良好集成（虽然机制与 WSL 1 不同）。
                - 缺点：
                  - 跨系统文件操作（从 Windows 访问 Linux 文件）性能不如 WSL 1，建议在 Linux 文件系统内进行项目开发。
                  - 需要虚拟化技术支持（现代 CPU 基本都支持）。

        - 区别
          - Cygwin
            - 核心思想：提供一个在 Windows 上原生运行的 POSIX 兼容层。
            - 它通过一个名为 cygwin1.dll 的动态链接库，将 POSIX 系统调用（如 fork, open, read）翻译成等价的 Windows API 调用。
            - 你在 Cygwin 下编译的源代码，不需要修改，就可以直接编译运行，但它依赖于 cygwin1.dll。
            - core = cygwin1.dll
            - 生成依赖 cygwin1.dll 的 Cygwin 程序。
          - WSL
            - 核心思想：在 Windows 内部运行一个真正的 Linux 内核。
            - WSL1：一个翻译层，将 Linux 系统调用直接转换为 Windows NT 内核调用。文件 I/O 性能初期较差。
            - WSL2：基于 Hyper-V 技术的轻量级虚拟机，运行一个完整的 Linux 内核。提供了完整的系统调用兼容性和极高的性能。
            - 你是在运行一个真正的 Linux 发行版（如 Ubuntu、Debian）。
            - core = Hyper-V 技术加持之下的轻量级虚拟机
          - MSYS2
            - 核心思想：为在 Windows 上编译和运行原生 Windows 程序提供一个构建环境。
            - 它基于 Cygwin 和 Arch Linux 的 Pacman 包管理器。
            - 它提供了 Bash shell 和一系列 GNU 工具，但其主要目标是生成不依赖 msys-2.0.dll 的原生 Windows 程序（如 .exe 和 .dll）。
            - 它是 MinGW-w64 的升级版和增强版。
            - core = cygwin + pacman包管理器

            - MinGW-w64 是一个自由开源的软件开发环境，用于创建微软 Windows 操作系统下的应用程序。它的核心是一个编译器套件，能够将用 C、C++、Fortran 等语言编写的源代码，编译成可在 Windows 32位 和 64位 平台上运行的原生可执行文件（.exe 和 .dll）。
            - 关键点在于 “原生” 这个词。MinGW-w64 生成的程序不依赖于额外的虚拟环境或模拟层（如 Cygwin）。它们直接调用 Windows 操作系统提供的 API（主要是 Win32 API 和 Win64 API）来运行，因此其运行效率和与系统的集成度都非常高。

        - 前身：MinGW（Minimal GCC for Windows）

        - ver.
          - MinGW： 原名 Minimal GNU for Windows。
            - 它起源于早期的一个名为 Cygwin 的项目。Cygwin 的目标是在 Windows 上提供一个类 Unix 的兼容层，让 Unix/Linux 程序可以很容易地移植到 Windows 上编译运行，但编译出的程序需要依赖 cygwin1.dll 这个库。
            - MinGW 从 Cygwin 中分支出来，其理念是摆脱对 Cygwin 运行时库的依赖，直接生成纯粹的原生 Windows 程序。它主要针对 32位（x86）Windows 平台。
          - MinGW-w64： 顾名思义，是 MinGW 的 64位 扩展分支。
            - 历史原因： 在 64位 Windows（x86-64/AMD64）出现后，原始的 MinGW 项目在支持 64位编译方面进展缓慢。
            - 分叉诞生： 一位名叫 OneVision 的开发者（据说是来自中国的尹首一）由于等不及官方支持，便自己创建了一个分支，为 MinGW 添加了对 64位架构的编译支持。这个分支后来发展成了独立的项目，即 MinGW-w64。
            - 现状： 如今，MinGW-w64 项目已经极大地超越了他的前身。它不仅支持编译 32位 和 64位 程序，还提供了更多的新功能和 API 支持（如更新的 C++ 标准、ARM 架构支持等），在活跃度和功能完整性上都已经成为了事实上的标准。许多流行的发行版（如 MSYS2 中集成的）实际上都是基于 MinGW-w64。

        - 1. 集中式工作流（Centralized Workflow）
          - 这是最基础的工作流，适合刚从 SVN 等集中式版本控制系统迁移过来的小团队或个人项目。

核心：只有一个中央仓库，一个主干分支（main/master）。所有开发者直接向这个分支提交代码。

流程：

git clone 克隆中央仓库。

在本地 main 分支上进行修改和提交。

在推送前，先 git pull 拉取远程最新的变更（解决可能的冲突）。

git push 推送到中央仓库。
        - 功能分支工作流（Feature Branch Workflow）
          - 这是目前最流行、最推荐的中小团队工作流，是 基于主干的开发 的典型实践。

核心：任何新功能或修复都必须在新的独立分支上完成，完成后通过 Pull Request（PR）或 Merge Request（MR）合并回主干。

流程：

基于最新的 main 分支创建一个新功能分支，例如 git checkout -b feature/user-authentication。

在新分支上进行开发、提交。

开发完成后，将分支推送到远程仓库：git push origin feature/user-authentication。

在 Git 平台（如 GitHub, GitLab）上创建一个 Pull Request，请求将 feature 分支合并到 main 分支。

团队成员在 PR 上进行代码审查（Code Review），提出修改意见。

根据审查意见修改代码，再次推送。

审查通过后，由项目维护者（或自己）合并 PR，并删除功能分支。

非连续：服务通常用于执行一次性的、需要明确结果的任务，而不是连续的数据流。

RCL： 这是一个用C语言编写的库，它定义了一套统一的ROS客户端API。rclpy 并不是直接与DDS交互，而是通过调用RCL的C函数。RCL负责将不同语言的调用转换为底层中间件的操作。这保证了ROS 2核心功能在不同语言客户端中的一致性。

    - 自主导航
      - def
        - 自主导航 是指机器人在没有人工干预的情况下，从一个起始点安全、高效地运动到目标点的能力。这不仅仅是“走过去”那么简单，它包含了三个核心问题：

我在哪？ - 定位

我要去哪？ - 目标点

我如何过去？ - 路径规划与控制
      - content
        - 感知： 通过传感器（如激光雷达、摄像头、IMU、里程计）来感知环境和自身状态。
        - 地图构建： 利用感知数据（尤其是激光雷达）构建环境地图，常用的是2D栅格地图。
        - 定位： 在地图中确定机器人当前的位姿（位置和朝向）。AMCL 是ROS中经典的2D定位算法。
        - 路径规划
          - 全局规划器： 根据地图、目标点和机器人当前位置，计算出一条从起点到终点的全局最优或次优路径。
          - 局部规划器： 根据全局路径和实时感知的障碍物信息，计算机器人实际的线速度和角速度指令，以跟踪全局路径并避开未知或动态障碍物。
        - 控制： 将局部规划器计算出的速度指令发送给机器人的底盘控制器，驱动电机运动。
    - 行为树
      - def
        - 行为树是一种用于建模机器人决策和控制的树状结构。
在Nav2中，它用于编排导航任务（如等待、计算路径、恢复行为等），使其更加模块化、灵活和可维护。

与有限状态机相比，行为树的主要优点是高可复用性和更好的反应性。
      - content
        - 核心节点类型
          - 控制节点： 定义执行流程。
            - 序列： 按顺序执行所有子节点，如果任一子节点失败，则停止并返回失败。
            - 回退： 按顺序执行子节点，直到有一个成功为止。常用于尝试多种策略。
            - 并行： 同时执行所有子节点。
          - 执行节点： 实际执行任务的叶子节点。
            - 动作： 执行一个耗时较长的动作，如 ComputePathToPose。
            - 条件： 检查某个条件是否满足，如 IsBatteryLow。
            - 装饰器： 修改子节点的行为，如重复执行、超时设置等。
      - tool
        - 树仿真器Groot2
          - func
            - 图形化编辑： 通过拖放节点来构建和编辑行为树。
            - 实时监控： 在Nav2运行时，可以连接到ROS 2，实时查看节点的执行状态（成功、失败、运行中）。
            - 验证与调试： 帮助开发者直观地理解和调试复杂的导航逻辑。
    - 几个特定节点
      - ComputePathToPoseAction
        - 全局规划器节点。接收一个目标位姿，利用全局规划器算法（如NavFn， Smac）在地图上计算出一条全局路径。
      - FollowPathAction
        - 局部规划器/控制器节点。接收一条全局路径，结合实时传感器数据，输出速度命令来控制机器人沿着路径运动，并避开动态障碍物。
      - SpinAction / WaitAction
        - 特定行为节点。让机器人在原地旋转特定角度或等待一段时间。
      - BackUpAction
        - 恢复行为节点。让机器人向后移动一小段距离。
      - ClearEntireCostmap / ClearLocalCostmap
        - 清除代价地图节点。用于清除全局或局部代价地图中的障碍信息，例如当传感器出现短暂误报时。
      - RecoveryNode
        - 恢复节点。这是一个控制节点，它包裹了一组恢复行为（如清除代价地图、后退、旋转），如果主任务失败，它会按顺序执行这些恢复行为。

          - 行为服务器。托管特定行为动作，如Spin， Wait， Backup等。
        - BT Navigator Server
          - 行为树导航服务器。这是导航系统的“大脑”。它加载一个定义导航流程的行为树XML文件，并调用其他服务器（Planner, Controller, Behavior）的动作节点来完成导航任务。
        - Amcl
          - 自适应蒙特卡洛定位。为机器人提供在已知地图中的2D位姿估计。它是一个独立的节点，但与其他模块紧密协作。
        - Map Server
          - 地图服务器。提供静态地图（通常是occupancy_grid）给其他模块使用。
        - Costmap 2D
          - 代价地图。分为Global Costmap（用于全局规划）和Local Costmap（用于局部规划和避障）。它将传感器数据、静态地图和障碍物信息融合成一个网格地图，其中每个单元格都有一个代价值，表示通过该区域的难度。

  - logic
    - 细分主题 1
    - 目标接收： BT Navigator 接收到一个新的目标位姿。
    - 行为树触发： BT Navigator 开始执行其加载的默认行为树。
    - 全局路径规划： 行为树首先触发 ComputePathToPose 动作。该动作请求 Planner Server。Planner Server 结合 AMCL 提供的当前位姿、目标位姿以及 Global Costmap 的信息，计算出一条全局路径，并返回给行为树。
    - 路径跟踪与局部避障： 行为树随后触发 FollowPath 动作。该动作请求 Controller Server。Controller Server 接收全局路径，并结合 Local Costmap 的实时障碍物信息，计算出具体的速度指令（cmd_vel）发送给机器人底盘，驱动机器人运动。
    - 循环与监控： 在跟踪路径过程中，Controller Server 会持续监控机器人的位姿和周围环境。行为树也会持续监控 FollowPath 节点的执行状态。
    - 成功或恢复：
      - 成功： 如果机器人到达目标点附近且满足容差要求，FollowPath 节点返回成功，整个导航任务完成。
      - 失败/恢复： 如果机器人被卡住（例如，路径被临时阻塞），FollowPath 节点会返回失败。此时，行为树中的 RecoveryNode 会被触发。它会按预定义的顺序执行恢复行为（例如，先ClearLocalCostmap，再Spin，最后BackUp），尝试让机器人摆脱困境。恢复行为成功后，流程会跳回第3步或第4步，重新尝试规划或跟踪。
    - 取消： 在导航过程中，如果接收到一个新的目标或取消命令，当前任务会被中止，系统进入空闲状态，等待下一个目标。
